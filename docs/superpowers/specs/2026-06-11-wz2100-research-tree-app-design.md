# WZ2100 Research Tree — дизайн

Дата: 2026-06-11. Статус: затверджено користувачем.

## Мета

Веб-апка для Warzone 2100 (скірміш/мультиплеєр), яка:

1. показує все дерево досліджень (390 технологій) графічно, з іконками;
2. будує план дослідження від обраних цілей;
3. дає відмічати прогрес вручну під час гри (лайв-режим).

Мова: інтерфейс українською, назви технологій англійською (як у даних гри).

## Поза скоупом v1

- Автосинхронізація з грою (мод/експорт стану). Архітектурно не блокуємо: лайв-сесія — окремий стор, до якого потім можна додати інше джерело подій.
- Оцінка часу досліджень (ETA по кількості лабораторій) — тільки вартість у research points.
- Дерево кампанії — тільки `data/mp/stats`.
- Рендери 3D-моделей юнітів — тільки категорійні іконки з атласу інтерфейсу.
- Українські назви технологій з .po-файлів.
- Бекенд, акаунти, синхронізація між пристроями.

## Стек

Статична SPA: **Vite + Vue 3 (Composition API, TypeScript) + Pinia + Vue Flow + elkjs (тільки в build-скрипті)**. Тести — Vitest. Проєкт: `~/CODING/wz2100-tree`. Деплой за бажанням — GitHub Pages (поза скоупом v1).

## Дані

Джерело: `data/mp/stats/research.json` з репозиторію Warzone2100/warzone2100, **зафіксовано на тег `4.7.0`** (останній стабільний реліз, 2026-04-07).

Build-скрипт `scripts/build-data.ts` (запускається вручну через `npm run build:data`):

1. Завантажує research.json за тегом.
2. Нормалізує кожен запис: `id`, `name`, `points` (researchPoints), `branch` (виводиться з iconID/category: зброя, захист, системи, конструкції, …), `subgroup` (subgroupIconID), `prereqs` (requiredResearch), `unlocks` (зворотні ребра), `results` (resultComponents/resultStructures — для панелі деталей).
3. Валідує граф: ациклічність, усі prereq-id існують; при помилці скрипт падає з поясненням.
4. Рахує розкладку через elkjs (layered, зліва направо) і записує готові `x/y` для кожного вузла.
5. Пише `src/data/research-4.7.0.json` (вузли + координати + метадані версії).

Іконки: категорійні спрайти витягуються з атласу інтерфейсу гри (GPL, з зазначенням джерела) у `public/icons/`, ім'я файлу = iconID/subgroupIconID. Якщо для якогось iconID спрайт не витягнувся — fallback на загальну іконку гілки; build-скрипт виводить список таких.

## UI: розкладка «три панелі» (обрана з мокапів, варіант A)

```
┌──────────────────────────────────────────────────────┐
│ Топбар: назва · 🔍 пошук · тумблер «▶ Гра»            │
├────────┬───────────────────────────────┬─────────────┤
│ Гілки  │ Дерево (Vue Flow, пан/зум)    │ Деталі /    │
│ (фільт-│ вузли = картки з іконками     │ План /      │
│  ри)   │                               │ Гра-прогрес │
└────────┴───────────────────────────────┴─────────────┘
```

### Дерево

- Пан/зум; рендер лише видимих вузлів (`onlyRenderVisibleElements`).
- Вузол: іконка, назва, вартість. Рамка за станом:
  - **вивчено** (зелена) — у лайв-сесії відмічено;
  - **доступно** (жовта) — всі prereq вивчені;
  - **у плані** (синя) — входить в активний план;
  - **недоступно** (сіра) — решта.
  Стани комбінуються: «у плані + доступно» — синя рамка з жовтою позначкою.
- Пошук: інкрементальний список збігів → клік → камера перелітає до вузла, вузол блимає.
- Фільтри гілок (чекбокси зліва): вимкнена гілка притемнюється (opacity), не приховується — розкладка стабільна.
- Hover або вибір вузла підсвічує повний ланцюжок пререквізитів.
- Клік → панель деталей: опис/назва, вартість, пререквізити (клікабельні — перелітають), що відкриває (компоненти/структури/дослідження), кнопки «У план» / «Вивчено» (друга — тільки в режимі гри).

### Планувальник

- «У план» на цілі → автоплан: топологічне сортування об'єднаного підграфа пререквізитів усіх цілей, без дублікатів, вже вивчене в сесії виключається.
- Черга в правій панелі: drag&drop переставляння; дозволені лише позиції, що не порушують порядок пререквізитів — недопустима позиція відхиляється з підсвіткою конфліктного пререквізиту.
- Видалення цілі прибирає з черги її пререквізити, не потрібні іншим цілям.
- Кілька іменованих планів, перемикач; активний план підсвічується на дереві.
- Експорт/імпорт плану як JSON-файл.

### Режим гри (лайв)

- Тумблер у топбарі; не окремий екран — стан поверх дерева.
- Відмічання вивченого: клік по вузлу (у режимі гри) або по пункту черги.
- Зняти позначку можна (місклік) — стани залежних вузлів перераховуються.
- Права панель у режимі гри: прогрес активного плану (n/m, %), наступні доступні пункти черги зверху, підсвічені.
- «Нова гра» скидає сесію (з confirm).

## Стан (Pinia)

- `useDataStore` — статичний граф (вузли, ребра, індекси prereq/unlocks). Завантажується з імпортованого JSON, не змінюється.
- `usePlansStore` — іменовані плани: `{ id, name, goals[], queue[] }`, активний план. Автоплан і валідація drag&drop — чисті функції в `src/lib/planner.ts`.
- `useSessionStore` — лайв-сесія: `Set<researchId>` вивченого. Обчислювані стани вузлів — derived з session + plans.

Persistence: plans і session → localStorage, ключі з версією схеми (`wz2100-tree:v1:plans`). Несумісна/зіпсована схема → пропозиція скинути (confirm), не мовчазний краш.

## Помилки

- Дані валідуються в build-скрипті — рантайм довіряє JSON.
- localStorage: читання через try/catch; при помилці парсингу — діалог «збереження несумісне, скинути?».
- Імпорт плану: валідація id проти графа; невідомі id — відмова з переліком.

## Тести (Vitest)

Чиста логіка, без рендеру графа:

- `planner.ts`: автоплан від однієї/кількох цілей (порядок валідний, без дублікатів, вивчене виключено), видалення цілі, валідація позицій drag&drop.
- `nodeState.ts`: обчислення станів вузлів (вивчено/доступно/у плані/недоступно) на малому фікстурному графі.
- `storage.ts`: round-trip персистенсу, обробка зіпсованого JSON, несумісної версії.
- build-скрипт: юніт на нормалізацію і виявлення циклів (фікстури), без мережі.

Компонентні/E2E тести — поза скоупом v1; графіка перевіряється вручну.
